# Crash Risk Page Design

**Date:** 2026-04-10  
**Owner:** Codex + 项目维护者  
**Scope:** 新增独立页面 `/crash-risk`，为 `leaderboard` 候选池提供“结构性崩盘预警”评分与解释

---

## 1. 背景与目标

当前项目已经具备三类成熟能力：

1. `leaderboard`：筛出短周期强势币。
2. `rebound`：衡量多周期反弹幅度。
3. `risk`：承载账户与持仓风险提示。

本次新增功能不做“是否必崩”的二元判断，而是针对 **最新 `leaderboard` 候选池**，识别“价格仍强，但结构开始恶化”的标的，并以独立页面方式输出：

1. **结构背离强度**
2. **触发阶段**
3. **风险评分**
4. **触发原因**

目标是让用户快速定位“最值得重点盯的强势币”，用于盘前/盘中研究，而不是替代交易决策。

---

## 2. 产品定位

页面名称建议：**结构性崩盘预警**

页面路径建议：`/crash-risk`

定位：

1. 这是一个 **市场研究页**，不是账户汇总页。
2. 这是一个 **候选池排序页**，不是全市场扫描器。
3. 这是一个 **风险提示页**，不是自动交易信号页。

候选池约束：

1. 仅扫描 **最新一期 `leaderboard` 快照中的 `symbol`**。
2. 不在第一版做全市场遍历。
3. 不在第一版引入用户持仓方向耦合。

---

## 3. 核心模型

### 3.1 模型定义

模型目标：识别“高位拉抬或高位维持过程中，仓位结构、动能和承接逐步变脆”的标的。

叙事基础来自以下市场结构：

1. 价格继续上行或维持高位。
2. `1H OI` 连续下降，显示新增杠杆参与不足或高位出清。
3. 后半段动能减弱，价格推进效率下降。
4. 高位承接转差，一旦失守短周期关键位更容易进入加速下跌。

该模型输出的是 **风险程度**，不是确定性的方向判断。

### 3.2 分层输出

页面统一输出三层：

1. `观察`：价格上行，但 `1H OI` 开始走弱。
2. `警惕`：价格继续抬升，`OI` 连续 `3-5` 根下降。
3. `高危`：价格创新高或贴近高位，同时 `OI`、成交额、后续动能同步转弱。

---

## 4. 评分框架

总分：`0-100`

分数组成：

1. `OI 背离分`：`0-35`
2. `价格扩张分`：`0-20`
3. `动能衰减分`：`0-20`
4. `承接脆弱分`：`0-15`
5. `触发确认分`：`0-10`

### 4.1 OI 背离分（0-35）

用于衡量“价格强、OI 弱”的程度。

建议规则：

1. 近 `6H/12H` 价格净涨幅为正，且 `1H OI` 净变化为负时进入计分。
2. `OI` 连续下降根数越多，分越高。
3. 价格涨幅越大而 `OI` 跌幅越深，分越高。
4. 若价格位于日内高位附近出现该背离，追加权重。

### 4.2 价格扩张分（0-20）

用于衡量“上涨是否已经过度”。

建议规则：

1. 统计近 `6H/12H/24H` 涨幅。
2. 涨幅越异常，分越高。
3. 若该币已在 `leaderboard` 中排名靠前，可作为轻微增强项。

### 4.3 动能衰减分（0-20）

用于衡量“后半段推进效率是否下降”。

建议规则：

1. 最近几根 `1H` 收盘涨幅变小。
2. 价格回撤幅度扩大。
3. K 线收盘位置偏离高点的程度增加。
4. 创新高效率下降但震荡增多。

### 4.4 承接脆弱分（0-15）

用于衡量“高位是否缺乏稳固承接”。

建议规则：

1. 高位横盘或缓升，但成交额未同步增强。
2. 高位停留时间变长，推进幅度不足。
3. 价格贴近高点，但结构承接恶化。

### 4.5 触发确认分（0-10）

用于衡量“是否从风险提示进入验证阶段”。

建议规则：

1. 跌破短周期支撑或最近波段低点时加分。
2. 最近 `1-2` 根 `1H` 出现明显转弱时加分。
3. 高位横盘后的首次放大回撤可作为额外确认信号。

### 4.6 总分分级

1. `0-39`：观察
2. `40-69`：警惕
3. `70-100`：高危

## 5. 页面信息架构

页面分为四个区域：

### 5.1 顶部摘要区

显示：

1. 当次扫描时间
2. 候选池数量
3. `高危 / 警惕 / 观察` 数量
4. 手动刷新按钮

### 5.2 风险榜单区

默认按 `Risk Score` 降序展示。

每行字段建议：

1. `Symbol`
2. `Risk Score`
3. `Stage`
4. `结构背离强度`
5. `Drivers`
6. `24H 涨幅`
7. `近 N 小时 OI 变化`
8. `Source Badge`（例如 `Leaderboard`）

### 5.3 单币详情区

点击榜单行后展开：

1. 价格走势
2. `1H OI` 走势
3. 成交额/成交量走势
4. 各子分项分值
5. 最近一次触发确认原因
6. 候选池来源说明（对应 `leaderboard` 快照时间）

### 5.4 方法说明区

补充说明：

1. 本页输出的是结构风险，不代表确定性做空信号。
2. 候选池仅来源于最新 `leaderboard` 快照。
3. 模型采用规则化评分，强调可解释性与可调参数。

---

## 6. 数据输入与处理

### 6.1 候选池来源

从最新一期 `leaderboard` 快照读取 `symbol` 列表。

### 6.2 必需市场数据

针对候选池中的每个 `symbol`，拉取：

1. 当前价
2. 近 `6H/12H/24H` 涨幅所需的 `1H` K 线
3. `1H OI` 时间序列
4. `1H` 成交额或成交量时间序列
5. 近 `N` 根 `1H` K 线的 `high/low/close`

### 6.3 增强字段

第一版可选增强：

1. `leaderboard` 排名
2. 是否同时出现在 `rebound` 榜单
3. 与日内高点距离

---

## 7. 系统设计

第一版采用 **快照读取 + 手动刷新重算**，不做高频实时流式计算。

建议新增以下模块：

1. `app/services/crash_risk_service.py`
2. `app/api/crash_risk_api.py`
3. `templates/crash_risk.html`
4. `static/js/crash-risk.js`

若需要持久化快照，建议后续增加：

1. `app/repositories/crash_risk_repository.py`
2. 新快照表（或复用现有快照模式）

第一版可先采用：

1. 页面读取最新计算结果
2. 手动点击触发重算
3. 后续再引入定时任务生成晨间快照

---

## 8. API 设计建议

建议至少提供两个接口：

1. `GET /api/crash-risk`
   - 返回最新榜单结果与摘要
2. `POST /api/crash-risk/refresh`
   - 手动触发候选池重算

返回结构建议包含：

1. `as_of`
2. `source_snapshot`
3. `summary`
4. `rows`

每个 `row` 建议包含：

1. `symbol`
2. `risk_score`
3. `stage`
4. `divergence_strength`
5. `drivers`
6. `price_change_6h`
7. `price_change_12h`
8. `price_change_24h`
9. `oi_change_n_hours`
10. `component_scores`

---

## 9. 约束与非目标

### 9.1 约束

1. 第一版只扫描 `leaderboard` 快照中的 `symbol`。
2. 第一版优先保证解释性，不引入黑盒模型。
3. 第一版不耦合用户持仓，不做交易建议。
4. 第一版不追求秒级实时。

### 9.2 非目标

1. 不做全市场扫描。
2. 不做自动下单。
3. 不做复杂多图联动分析台。
4. 不将该评分直接用于账户风控熔断。

---

## 10. 风险与缓解

1. **误报偏多**  
   缓解：候选池仅限 `leaderboard`，并保留人工可调阈值。

2. **解释不足导致用户不信任**  
   缓解：每个币都输出 `drivers` 和 `component_scores`。

3. **数据抓取成本偏高**  
   缓解：只扫描快照候选池，不遍历全市场。

4. **页面和主面板职责混淆**  
   缓解：独立页面承载，不混入 `/` 主仪表盘。

---

## 11. Definition of Done

满足以下条件可视为设计落地完成：

1. 新页面 `/crash-risk` 可访问。
2. 页面默认读取最新 `leaderboard` 候选池。
3. 每个候选币输出 `Risk Score / Stage / Drivers`。
4. 评分拆分为 5 个可解释子分项。
5. 用户可在页面中查看单币明细与方法说明。
